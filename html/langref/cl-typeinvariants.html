1  Type invariants

There is a general recipe for constructing a list of parameters or properties "x1: T1{c1}, ..., xk: Tk{ck}" that must satisfy a given (satisfiable) constraint "c". 
class Foo(x1: T1{x2: T2; ...; xk: Tk; c},
          x2: T2{x3: T3; ...; xk: Tk; c},
          ...
          xk: Tk{c}) {
  ...
}
The first type "x1: T1{x2: T2; ...; xk: Tk; c}" is consistent iff "$x1: T1, x2: T2, ..., xk: Tk. c" is consistent. The second is consistent iff 
"x1: T1{x2: T2; ...; xk: Tk; c}
$x2: T2. $x3: T3, ..., xk: Tk. c
But this is always true. Similarly for the conditions for the other properties.
Thus logically every satisfiable constraint "c" on a list of parameters "x1", ..., "xk" can be expressed using the dependent types of "xi", provided that the constraint language is rich enough to permit existential quantifiers.
Nevertheless we will find it convenient to permit the programmer to explicitly specify a depclause after the list of properties, thus: 
class Point(i: Int, j: Int) { ... }
class Line(start: Point, end: Point){end != start} { ... }
class Triangle (a: Line, b: Line, c: Line)
   {a.end == b.start,  b.end == c.start,
    c.end == a.start} { ... }

$x1: T1, ..., xn: Tn, self: C. c
is valid (always true).
The guard is an invariant on all instances of the class or interface.
