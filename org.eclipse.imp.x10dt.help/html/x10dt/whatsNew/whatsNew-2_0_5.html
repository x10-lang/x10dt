What's New in 2.0.5

Here are descriptions of some of the more interesting or significant changes made to the X10 Tooling Platform for the 2.0.5 release. They are grouped into:

  • X10 Development Toolkit
  • Java back-end
  • C++ back-end
  • Additional Notes
X10 Development Toolkit
Prevention of installation on unsupported platforms The following platforms are supported:
  • Linux
  • MacOS
  • Windows
on both x86 and x86_64 architectures. X10DT will prevent you from installing on other platforms.

NB: Note that this does not prevent end-users from using a supported platform for remote compilations and launching with C++ back-end.
Auto-Indent The source editor now provides auto-indent support after typing newline and tab.

Auto-Indent behavior has been improved for 2.0.5 and supports preferences controlling, such as indentation width, tab display width and whether to use spaces or tabs.
Java Back-End
Use of ECJ compiler instead of downstream JDT builder Previous versions of the X10DT placed generated Java files in the project's source folder. As a result one could see markers on these files which was potentially confusing.

In this release, generated Java files are now placed within the X10 project's output directory and are compiled there via an invocation of the ECJ compiler.

This is a consistent behavior with the C++ back-end.
Search for main methods In a effort to share similar end-user experience between C++ and Java back-end, the X10 launch configuration Search button will now present all main X10 types within the project selected (or the workspace) rather than compiled Java main types.

C++ Back-End
Automatic detection of OS type and processor architecture Prior to 2.0.5 an automatic detection of the OS and processor characteristics were only detected for a local connection. In 2.0.5, we added a similar capability when creating a new remote connection.

When you click on the Add button, fill the required fields and finish with Validate.

We will automatically try to detect the right OS and processor architecture for the machine you have added.
Run as X10 application One can now select X10 application (C++ back-end) in the Run as context menu.
Mix of C++ native files with X10 files The X10 @Native construct is now supported and so one can have C++ files and X10 files in the same project.

This feature works locally and remotely.
Automatic check for required compiler version We have added a version check for the various C++ compilers that X10 supports.

Additional Notes

Memory Consumption

Memory consumption by the editor and the build process has been improved since previous releases. It is still a work in progress. You may encounter either two memory-related errors:
  • Java Heap Space OutOfMemory: You can increase the maximum heap size by specifying the -Xmx (Sun VM) VM option in the eclipse.ini in your Eclipse installation. For more details, see the Eclipse online help.
  • Java PermGen OutOfMemory: You can increase the PermGen area's size by specifying the -XX:MaxPermSize (Sun VM) in the eclipse.ini file.
Additionally, it may be useful to reduce the number of simultaneously opened editors.

Builder Performance

The current build mechanism is somewhat conservative in determining the set of files to be recompiled upon each save. This will cause longer build times under certain circumstances.

We've provided a building option, "Conservative Build" - turned on by default, which if turned off (see X10 Preferences) can result in shorter build times. However the shorter build time comes at the expense of incorrect behavior for certain X10 source programs.
