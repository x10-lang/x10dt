X10 basics

Core Object-oriented features

The core object-oriented features of X10 <#234#><#234#> are very similar to those of Java or C#.

A program consists of a collection of top-level compilation units (``unit'' for short). A unit is either a <#235#>class<#235#>, a <#236#>struct<#236#> or an <#237#>interface<#237#>.
#xtennum239#

Classes, structs and interfaces live in <#242#>packages<#242#> (e.g.<#243#><#243#> examples above). Packages typically consist of a sequence of identifiers separated by ``<#245#>.<#245#>'' (with no spaces).

Packages are used to control access to top-level units. If they are marked <#246#>public<#246#> they can be accessed from code in any other package. If they have no accessibility modifier, they can be accessed only by code living in the same package.

A file can contain multiple units -- however only one of them may be marked <#247#>public<#247#>. The name of the file must be the same as the name of the <#248#>public<#248#> unit, with the suffix ``<#249#>.x10<#249#>''.

The <#251#>fully qualified name<#251#> (FQN) of a class, a struct or an interface is the name of the class, the struct or the interface prefixed with the name of the package that unit lives in. A unit <#253#>A<#253#> must use the FQN of another unit <#254#>B<#254#> unless it has an explicit <#255#>import<#255#> statement.

Class

A class is a basic <#266#>bundle<#266#> of data and code. It specifies a number of <#267#>members<#267#>, namely <#268#>fields<#268#>, <#269#>methods<#269#>, <#270#>constructors<#270#>, and member classes and interfaces. Additionally a class specifies the name of another class from which it <#271#>inherits<#271#> (the <#272#>superclass<#272#>) and zero or more interfaces which it <#273#>implements<#273#>.

The members available at a class (i.e.<#274#><#274#> that can be used by variables whose type is that class) are those defined at a class, and those defined in superclasses.

A class may be marked <#275#>final<#275#>. <#276#>final<#276#> classes cannot be subclassed.

Fields may be mutable (var) or immutable (val). The type of a mutable field must always be specified (it follows the name of the variable, and a ``<#283#>:<#283#>''). A mutable field may or may not be initialized. The type of an immutable field may be omitted if the field declaration specifies an <#284#>initializer<#284#>. The value of an immutable field does not have to be specified through an initializer -- as long as it is supplied in every constructor for the class. However, if an initializer is not specified the type of the field must be specified.

Fields may be instance or static. By default fields are instance fields, static fields are marked with the flag static (as illustrated above). Every object has one copy of an instance field. However, in each place, there is only one copy of each static field for all instances of that class in the place. In X10 <#288#><#288#> static fields are required to be immutable.

Instance fields are inherited by subclasses. This means that an object always has enough space for the instance fields of the class of which the object is an instance, and all of the object's superclasses.

A field defined in a class may be <#289#>shadowed<#289#> in a subclass if the subclass defines a field with the same name (but possibly a different type). The value of a shadowed field can always be accessed by using the qualifier super.

It is a compile-time error for a class to declare two fields with the same name.


Properties

A class may specify one or more properties. Properties are immutable instance fields that can be used in constrained types <#293#>sec:x10-basic-constrained-types<#293#>. They are also distinguished from other instance fields in that they are initialized in a particular way, through the invocation of a <#294#>property<#294#> call during execution of the constructor <#295#>sec:x10-basic-constructor<#295#> for the object or struct. It is a compile time error for a constructor to have a normal execution path which does not contain a <#296#>property<#296#> call.

To make the compile-time type-checking of constraints more tractable, X10 v~2.5.2 <#297#><#297#> requires that the types of properties are ``simpler'' than the type of the class or struct they occur in. Methods

A method is a named piece of code, parametrized by zero or more variables (the <#303#>parameters<#303#>). A method may be <#304#>void<#304#> -- in which case it returns no value and is usually called just for its effect on the store -- or it may return a value of a given type.

A method may have one or more type parameters; such a method is said to be <#305#>type generic<#305#>. A method may have a <#306#>method guard <#306#>: the guard may specify constraints on the type parameters.

The <#308#>signature<#308#> of a method consists of its name and the types of its arguments. A class may not contain two methods whose signatures are the same. Additionally, a method specifies a <#309#>return type<#309#>. and zero or more <#310#>checked exceptions<#310#> that the body of the method is permitted to throw. (The body of the method may throw any number of unchecked exceptions.)

A value <#311#>e<#311#> may be returned from the body of a method by executing a <#312#>return e;<#312#> statement. The return type of a method may be inferred in X10 <#313#><#313#> (that is, it does not need to be explicitly supplied by the user). It is the least upper bound of the types of all expressions <#314#>e<#314#> where the body of the method contains a <#315#>return e<#315#> statement.

Methods may be <#316#>instance<#316#> methods or <#317#>static<#317#> methods. By default methods are instance methods. Methods are marked static by using the qualifier <#318#>static<#318#>.

The instance methods of a class are available for every object that is an instance of the class. Instance methods are invoked using the syntax <#323#>e.m(e1,..., en)<#323#>. <#324#>e<#324#> is said to be the <#325#>receiver<#325#> of the method invocation, and <#326#>e1,...,en<#326#> the <#327#>arguments<#327#>.

Inheritance

Methods may be inherited. That is, methods defined on superclasses are available as methods on a subclass, unless they are overridden by another method declaration with the same signature. Instance methods are said to be <#360#>virtually dispatched<#360#>: this means that when a method is invoked on an object o that is an instance of class C, the inheritance tree for C is searched (starting from C) for a method definition with the same signature. The first definition found is executed. The type rules for X10 <#365#><#365#> guarantee that at runtime such a method will exist.

Overloading

In X10 <#367#><#367#> methods may be overloaded. This means that a class may have multiple methods with the same name -- of necessity they must have a different signature. These methods have nothing to do with each other.

Overloading is very convenient -- it permits the programmer to use a method name for some abstract concept and than provide instantiations of that concept for different parameter types through different overloaded methods with the same name. The name <#368#>search<#368#> is overloaded in Table~#table:NQueens-Board#369>, for instance.

Access control

The qualifiers <#371#>private, public, protected<#371#> may be used to limit access to a method. <#372#>private<#372#> methods may be accessed only by code in the same class. <#373#>public<#373#> methods can be accessed from any code. <#374#>protected<#374#> methods can only be accessed in the same class or its subclasses. If no qualifier is provided, a method can be accessed by code in the same package (it is said to be <#375#>package protected<#375#>).

Structs

An object is typically represented through <#399#>handles<#399#>, indirect references to a contiguous chunk of data on the heap. The space allocated for an object typically consists of space for its fields and space for a <#400#>header<#400#>. A header contains some bytes of data that represent meta-information about the object, such as a reference to the table of methods used to implement virtual dispatching on the object. X10 also permits the definition of <#401#>structs<#401#>, which are different.

A <#402#>struct<#402#> is a ``header-less'' object. It can be represented by exactly as much memory as is necessary to represent the fields of the struct (modulo alignment constraints) and with its methods compiled to ``static'' methods. This is accomplished by imposing certain restrictions on structs (compared to objects). A <#403#>struct<#403#> definition is just like a <#404#>class<#404#> definition except that it is introduced by the keyword <#405#>struct<#405#> rather than by <#406#> class<#406#>. Like a class, a struct may define zero or more fields, constructors and methods and may implement zero or more interfaces. However, a struct does not support inheritance; it may not have an <#407#>extends<#407#> clause. A struct may not be recursive: that is, there can be no cycles in the graph whose nodes are struct definitions and whose edges from #tex2html_wrap_inline687# to #tex2html_wrap_inline689# record that the struct #tex2html_wrap_inline691# has a field of type (struct) #tex2html_wrap_inline693#.

X10 <#417#><#417#> does not support any notion of a reference to a struct. Structs are passed by value. Equality == on structs is defined as point-wise equality on the fields of the struct. Further, structs are immutable; no syntax is provided to update fields of a struct in place. While these decisions simplify the language, they make it awkward to express certain kinds of code, hence mutability and references to structs may be introduced in future versions of the language.

Statements

The sequential statements of X10 <#446#><#446#> should be familiar from Java and C++.

Assignment.

The statement <#448#>x=e<#448#> evaluates the expression <#449#>e<#449#> and assigns it to the local variable <#450#>x<#450#>.

The statement <#451#>d.f=e<#451#> evaluates <#452#>d<#452#> to determine an object <#453#>o<#453#>. This object must have a field <#454#>f<#454#> that is mutable. This value is changed to the value obtained by evaluating <#455#>e<#455#>.

In both cases the compiler checks that the type of the right hand side expression is a subtype of the declared type.2.1

Conditionals.

<#458#>if (c) then S else S<#458#>. The first branch is executed if the condition <#459#>c<#459#> evaluates to true, else the second branch is executed. One-armed conditionals are also permitted (the <#460#>else S<#460#> may be dropped).

While loops.

<#462#>while (c) S<#462#>. The condition <#463#>c<#463#> is evaluated. If it is true, the body <#464#>S<#464#> is executed, and the control returns to the top of the loop and the cycle of condition evluation and body execution repeats. Thus the loop terminates only when the condition evaluates to false.

A while loop may be labeled <#465#>l: while (c) S<#465#> (just as any other statement). The body of a <#466#>while<#466#> loop may contain the statements <#467#>continue l;<#467#> and <#468#>break l;<#468#>. The first causes control to jump to the top of the loop labeled <#469#>l<#469#>, i.e. the rest of the code in the body of the loop following the <#470#>continue l;<#470#> statement is not executed. <#471#>break l;<#471#> causes control to jump after the loop labeled <#472#>l<#472#> (that is, control exits from that loop).

Throw statement.

X10 <#532#><#532#> has a non-resumptive exception model. This means that an exception can be thrown at any point in the code. Starting with X10 <#533#><#533#> 2.3.0, exceptions may be either <#534#>checked<#534#> or <#535#>unchecked<#535#> to more closely match the Java programming model for exceptions. Checked exceptions will be subclasses of x10.lang.CheckedException; if a method throws such an exception then it must declare that it does so with a throws clause (placed after the return type of the method, if any, and the body of the method).

Throwing an exception causes the call stack to be unwound until a catcher can be found (see below). If no catcher can be found the exception is thrown to the closest dyamically enclosing finish surrounding the throw point.

Try catch statement.

A try/catch/finally statement works just as in Java. It permits exceptions to be caught (through a <#541#> catch<#541#> clause) and computation resumed from the point of capture. A finally clause specifies code that must be run whenever control exits the try block -- regardless of whether the return is normal or exceptional.

Return statement.

The statement return; causes control to return from the current method. The method must be a void method. The statement return e; causes control to return from the current method with the value e.

A description of the other control constructs may be found in [#x10-lang-spec##1###].


Types

X10 <#553#><#553#> is a statically type-checked language. This means that the compiler checks the type of each expression, and ensures that the operations performed on an expression are those that are permitted by the type of the expression.

The name C of a class or interface is the most basic form of a type in X10. X10 <#555#><#555#> also supports

  • <#557#>generic types<#557#> (e.g.<#558#><#558#> Array[Int]) that take types as parameters,
  • <#560#>constrained types<#560#> (e.g.<#561#><#561#> Array{self.rank==1}) that specify an additional constraint on the properties of the base type,
  • <#563#>nested types<#563#> (e.g.<#564#><#564#> MyArray[T].Reducer
  • <#566#>functional type<#566#> (e.g.<#567#><#567#> (Int)=;SPMgt;Int.

A type specifies a set of members (fields, methods, constructors) that are defined on expressions of that type.

The availability of members of a type also depends on the accessibility relation. That is, a member of a type may be defined but not accessible (e.g. it is marked private).

A variable of a type S can only be assigned values whose static type T is the same as S or is a <#574#>subtype<#574#> of S.

A type C is a subtype of type E if there is some type D such that C is a subtype of D and D is a subtype of E.

If (class or interface) C[X1,...,Xm] extends or implements type D[S1,..., Sn] (for m and n non-negative integers) then for every mapping A from type variables to types, the type obtained by applying A to C[X1,...,Xm] is a subtype of the type obtained by applying A to D[S1,...,Sn].

A constrained type C{c} is a subtype of D{d} if C is a subtype of D and c implies d.

A nested type A.B is a subtype of C.D if A is the same as C and B is a subtype of D.

A function type (A1,...,Am)=;SPMgt;B is a subtype of (D1,...,Dn)=;SPMgt;E if m=n, Di is a subtype of Ai (for i in 1..n) and B is a subtype of E.

Constrained types

Constrained types are a key innovation of X10.

A constrained type is of the form T{c} where T is a type and c is a Boolean expression of a restricted kind. c may contain references to the special variable self, and to any final variables visible at its point of definition.

Such a type is understood as the set of all entities o which are of type T and satisfy the constraint c when self is bound to o.

The permitted constraints include the predicates == and !=. These predicates may be applied to constraint terms. A constraint term is either a final variable visible at the point of definition of the constraint, or the special variable self or of the form t.f where f names a field, and t is (recursively) a constraint term. In a term self.p p must name a property of self.
