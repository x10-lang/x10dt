Let's step through the program, a line at a time:
line 1:
// file HelloWorld.x10
Comments in X10 are the same as Java or C++: they either begin with "//" and go through the end of the line, or begin with "/*" and end at the first "*/" that follows. Because the first "*/" ends the comment, "/*...*/" comments do not nest.
X10 also uses the same comment conventions as JavaDoc for publishing APIs.
line 2:
class HelloWorld {... }
Like Java (and unlike C++) all X10 executable code lives in classes. Normally a class will have the same name as its file- HelloWorld is found in HelloWorld.x10.
line 3:
public static def main(argv: Array[String](1))
Program execution starts, as in Java and C++, with a method named main, which takes the command-line arguments as a collection of strings. But here we start to see some differences from Java and C++:
  • The keyword def begins a method declaration. This makes it easy to tell what's a method and what isn't.
  • You don't usually need to specify the return type for a methods. The compiler will discover the return statements in a method and will then normally be able to deduce the return type. If you do wish to specify it, then, unlike Java and C++, it follows the argument list. The return type for main is Void, meaning that it does not return a value. The compiler can easily tell that without your help. Note the `:' that separates the argument list from the return type: it is required.
    So why did we put the ": Void" there if we didn't have to? Basically, just to save someone other than us who is reading the code the trouble of figuring out what's returned. Keep in mind, too, that things are not always so simple as our main: the compiler will patiently read the method body, but do you want to force your human audience to do so?
  • X10 has generic types, along the same general lines as Java and C++. X10 uses square brackets to hold the actual type, e.g. Array[String] for declaring an array of Strings.
  • X10 arrays, like FORTRAN arrays, may be multi-dimensional. The "(1)" that follows Array[String] asserts that the array is one-dimensional, or in other words, is just like the usual Java or C++ array.
  • The general syntax for assigning a type to an identifier is
    identifier: type,
    as in argv:Array[String](1). White space before or after the `:' is ignored by the compiler. In fact, white space in X10 is treated as it is in Java and C++: normally ignored, except to the extent that it is needed to separate tokens or appears in string literal constants.
line 4:
x10.io.Console.OUT.println("Hello, World");
X10's libraries are arranged in packages, much like those of Java. The x10.io package contains many input-output classes.
Console.IN, Console.OUT, and Console.ERR
are the standard input, output, and error streams. The method println prints a string, followed by an operating-system dependent line-ender: either a single newline character for Unix based systems, or a carriage return-newline pair. If you don't want the newline, use print instead.
You can compile and run the Hello,World program as follows: 
% x10c HelloWorld.x10
% x10 HelloWorld
Hello, World
This uses a Java-based runtime to execute the program. There is also a C++ runtime. To use it, compile with x10c++ rather than x10c: 
% x10c++ HelloWorld.x10 -o hello
% runx10 hello
Hello, World
You'll need the runx10: you cannot invoke hello directly, because some special setup is required to accommodate the concurrency and the possible distribution of the execution across multiple machines. We don't actually need anything special here, but the code for booting an X10 program is generic.
